## What "healthy" looks like (golden signals)
- `GET /api/health` → 200 (liveness, no DB).
- `GET /api/health/ready` → 200 with `db.state == "connected"` and `ai.ok` true/skipped (readiness, DB- **and** AI-aware).
- App Gateway backend health = **Healthy**; `UnhealthyHostCount == 0`.
- App Insights: low/zero `requests` with `success==false`; no `SQL`/AI failed `dependencies`; no `ConnectionError` exceptions.
- New earnings rows appearing roughly every 10 min (scan cron `*/10 * * * *`).

## Standard investigation workflow
1. **Acknowledge the trigger.** Note which alert/signal fired (most often `alert-appgw-unhealthy-backend`).
2. **Triage the two health endpoints** (instantly separates DB outage from app-down):
   - `/api/health` 200 + `/api/health/ready` 503 ⇒ **DB or AI outage** (read `db`/`ai` in the body → Scenario 1 or 3).
   - both fail / 502 everywhere ⇒ **app down or gateway** (Scenario 2/4).
3. **Check App Gateway backend health** to confirm Healthy/Unhealthy.
4. **Query App Insights** for the failing dependency/exception in the incident window.
5. **Determine the root cause** from the evidence and apply the **narrowest safe,
   reversible** remediation yourself (the runbook shows how to detect and verify, not what
   the fix is).
6. **Verify recovery** via the golden signals and confirm the alert auto-mitigated.
7. **Report** (format below).

## Direct Azure SQL access (SRE agent identity)
The SRE agent can reach and query Azure SQL **directly** for any SQL-server-related
issue, because the environment is provisioned so both the network path and the
identity are already in place:
- **Network:** the SRE agent's **outbound IP address is added to the Azure SQL
  server firewall**, so the agent can open a connection to the server without any
  additional firewall change.
- **Identity:** the SRE agent's **managed identity is provisioned as a contained
  database user in the `JBDB` database and granted `db_owner`**, so it authenticates
  with its Entra token — no SQL login/password.

Use this direct access to inspect live DMVs, confirm root cause, and verify recovery
first-hand rather than relying only on App Insights telemetry.

Granted role:
- **`db_owner`** — full control of the `JBDB` database (read, write, DDL, DMV/state
  inspection). This is broad by design so the agent can both diagnose **and**
  remediate SQL-side issues (e.g. add a missing index, update statistics, kill a
  blocking session) without a permissions blocker.

Guidance:
- **Prefer direct queries** against the SQL server / `JBDB` named in the **Environment**
  block above for live diagnosis (e.g. active sessions, blocking chains,
  `sys.dm_exec_requests`, `sys.dm_db_resource_stats` for CPU, index/stats health) —
  this is faster and more authoritative than waiting on telemetry ingestion. Do **not**
  hard-code the server name; always read it from the Environment block.
- Authenticate with the agent's managed identity (Entra token); never use SQL auth.
- **Flush the plan for the offending query only — never the whole cache.** When a bad
  or parameter-sniffed plan is the root cause, do **not** run `DBCC FREEPROCCACHE`
  with no argument (that clears **every** cached plan server-wide and forces a mass
  recompile — disruptive and far wider than the incident). Instead scope it to the
  single plan: identify the query's `plan_handle` (e.g. from `sys.dm_exec_requests` /
  `sys.dm_exec_query_stats` joined to `sys.dm_exec_sql_text`, or from Query Store,
  which is enabled here) and remove just that plan with
  `DBCC FREEPROCCACHE (0x<plan_handle>);`, or use
  `sys.sp_query_store_remove_plan @plan_id = <id>` (Query Store). This is the narrowest
  reversible action and lets SQL build a fresh plan for that one statement.
- **`db_owner` grants full read/write/DDL** — with it you *can* apply SQL-side
  remediations directly. Still follow the **Remediation policy** below: prefer the
  **narrowest, reversible** change, and treat destructive actions (dropping objects,
  deleting/truncating data, changes that alter application data or behaviour) as
  requiring human approval even though the role technically permits them.
- Direct SQL access does not change the approval rules for destructive or
  infrastructure actions.

## Known blind spots & gotchas (important)
- **AI is a critical dependency** — readiness (`/api/health/ready`) returns 503 after the AI model has been unreachable past a ~120s grace window, so a sustained AI outage marks the backend Unhealthy and trips **both** `alert-ai-connectivity-loss` and `alert-appgw-unhealthy-backend` (502s on all routes). Short blips are absorbed by the grace window; the local engine and the pre-first-probe startup window never block readiness. Within the grace window, data still serves and only AI enrichment is missing.
- **App Insights `-o table` can show empty** even when data exists — use JSON.
- **`CredentialUnavailableError` / `EnvironmentCredential` exceptions are noise** (normal `DefaultAzureCredential` chain fallthrough) — do not treat as a fault.
- **Serverless DB resume is transient** — a brief `connecting` state is tolerated by a 90s readiness grace; don't over-react to a single blip. Only sustained failures (~90s+) trip the alert.
- **Direct `*.azurewebsites.net` access returns 403 by design** (inbound restricted to the App Gateway subnet). Investigate through the FQDN.
- **Never enable public network access on Azure SQL without first verifying a Private Endpoint isn't already in place** — if one exists, leave public access disabled and investigate connectivity over the private endpoint instead.
- **Do not explicitly attribute incidents to chaos/fault testing.** Treat every signal as a real fault and report root cause/remediation on its technical merits. Repeatedly pointing out that an issue is "just chaos testing" is unhelpful — investigate and remediate as if it were production.

## Remediation policy
**Safe / auto-approved (reversible):** read-only investigation (logs, metrics, App
Insights / Log Analytics queries) and **narrow, reversible** configuration changes that
return a resource to its intended state, plus `azd deploy` (code only). Always be ready to
roll the change back.

**Requires human approval (do not auto-execute):** deleting any resource/database; scaling
SKUs/capacity; broad WAF rule changes; `azd provision` / `azd up` (infra changes);
network/identity topology changes; anything destructive or affecting other workloads.

**Restarting the App Service always requires user confirmation — even when the agent is
operating autonomously.** A restart is disruptive (drops in-flight requests, resets the
scan cron and connection pools, and incurs cold-start latency), so any agent (including the
SRE agent) must obtain explicit user permission before restarting `app-010726194448`. This
is never an auto-approved action.

**Before enabling public network access on Azure SQL:** verify whether a Private Endpoint
already exists for the SQL server. If one is present, do **not** enable public access —
keep traffic on the private endpoint and investigate connectivity through it instead.
**In case public network access needs to be enabled, any agent (including the SRE agent)
must obtain explicit user permission first — even when operating autonomously.** This is
never an auto-approved action.

Always choose the narrowest action that restores service and verify before closing.

## Reporting format (use for every incident)
1. **Summary** — one line: what failed, impact, current status.
2. **Timeline** — detection time, key findings, actions, recovery time.
3. **Root cause** — the specific dependency/error (quote the exception text).
4. **Evidence** — the queries run and the salient rows (counts, resultCodes).
5. **Remediation** — exact action taken (and whether approval was needed).
6. **Verification** — golden signals confirming recovery; alert auto-mitigated.
7. **Follow-ups** — prevention ideas (e.g., add App Insights 5xx log alert, Private Endpoint for SQL).
